# Gen API Docs

将项目 HTTP 接口整理到 `API接口文档/` 下的 API 文档文件夹内，并按模块拆分为多个 Markdown 文件（`XXX.md`，`XXX` 为模块名称）。

## 执行流程

### 1. 扫描路由与接口信息

递归扫描 `backend_api_python/app/routes/` 下所有 `.py` 文件，提取：

- Blueprint 名称与 `url_prefix`
- `@bp.route(...)` 或 `@bp.get/post/put/delete(...)` 的 path、methods
- `login_required`、`admin_required` 等权限装饰器（判定认证级别）
- 接口描述提取规则：优先使用注释（含 docstring），注释缺失或不充分时再根据函数真实逻辑含义补充描述；若注释与函数真实逻辑冲突，必须以函数真实逻辑含义为准
- 请求参数（path、query、body）及是否必填、默认值
- 响应结构（优先从 `success_response(...)`、字典返回结构推断）

按需补充类型来源：

- `backend_api_python/app/models/`
- `backend_api_python/app/schemas/`
- 路由文件顶部的模型与 schema 导入
- 路由直接调用的 service 返回结构（仅在必要时追踪）

### 2. 组织文档目录结构

在 `API接口文档/` 下创建（或复用）API 接口文档目录，例如：

- `API接口文档/`

目录内文档按模块拆分，模块名自动映射规则固定为“按 Blueprint”：

- 以接口所属 `Blueprint` 名称作为模块名唯一来源
- Blueprint 名称规范建议：以代码定义书写为准（区分大小写），不做额外风格归一化
- 同一 `Blueprint` 下的全部接口写入同一个模块文件
- 模块文件命名：`{Blueprint名称}.md`
- 若 `Blueprint` 名称包含不适合作为文件名的字符，需做文件名安全转换（仅转换文件名，显示名称仍保留原 Blueprint 名称）
- 若出现重名 `Blueprint`（极少见），以 `url_prefix` 追加区分后缀，确保文件名唯一

- `API接口文档/user.md`
- `API接口文档/order.md`
- `API接口文档/strategy.md`

并额外生成总览索引文件：

- `API接口文档/README.md`

### 3. 计算接口变更

按 `method + full_path` 对比新旧接口集合，输出四类（按模块归档）：

- 新增接口
- 移除接口
- 修改接口（同 path+method 但参数/返回/认证/描述变化）
- 未变化接口

### 4. 生成模块化文档

输出到 API 文档目录内的模块文件（`XXX.md`），每个模块一个文件。结构建议：

1. 模块标题（模块名）
2. 模块内变更说明（新增/移除/修改统计与明细）
3. 模块接口详情
4. 模块接口索引表

同时更新 `README.md`，汇总：

1. 全局变更统计（新增/移除/修改）
2. 模块文档导航（模块名 + 文件路径）
3. 全量接口总索引（可选）

接口详情至少包含：

- 方法、路径、认证级别、描述
- 请求参数表（名称/类型/位置/必填/默认值/描述）
- JSON 请求体示例（如适用）
- 成功响应示例（200）
- 常见失败响应示例（401/403/500）

### 5. 最终输出给用户

完成后明确告知：

- 生成的文档目录路径
- 本次更新的模块文件列表（`XXX.md`）
- 新增/移除/修改接口数量（可含模块维度统计）
- 若为首次生成，标记“首次文档，无历史版本可比较”

## 约束要求

- 优先保证路径、方法、认证信息准确，描述可简洁但不得臆造。
- 接口描述必须准确，默认遵循“注释 > 函数实际含义”；但若出现冲突，必须以函数真实逻辑含义为准。
- 对无法静态推断的字段，标记“待确认”，不要虚构类型。
- 保持中文输出，表格字段命名风格统一。
- 严禁继续输出单一超大文档，必须按模块拆分为 `XXX.md`。
- 模块名映射必须按 Blueprint 执行，不再使用业务名或文件名作为主映射依据。
